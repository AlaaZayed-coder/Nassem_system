# نظام النسيم إخوان — نظرة شاملة على المشروع

نظام إدارة موارد (ERP) مبني خصيصاً لمصنع "النسيم إخوان" لتصنيع أبواب الرول، يغطي دورة العمل كاملة: من استلام الطلبية، عبر المبيعات والإنتاج والتركيب، وصولاً لتأكيد العميل — مع فروع جانبية للمشتريات والصيانة والمخزون.

## التقنية

- **Next.js 14** (App Router) + **TypeScript**
- **Supabase** (Postgres + PostgREST + Storage) — قاعدة بيانات حية واحدة، بدون بيئة staging
- **Tailwind CSS** للتنسيق
- **بوت تيليجرام** (وليس واتساب) كقناة تواصل رسمية مع الموظفين والفريق الميداني
- النشر عبر **Vercel**، يُبنى تلقائياً عند كل push لفرع `main`

### قيود معروفة ومتعمّدة
- **لا يوجد نظام تسجيل دخول حقيقي لكل موظف.** كل الصفحات تُفتح بدون مصادقة؛ "الدور" (`erp_staff.role`) يُستخدم فقط للتصنيف والعرض (مثل صفحة الأجندة)، وليس كضابط صلاحيات فعلي. "علاء زايد" اسم مثبّت في رأس الصفحة حالياً كمسؤول النظام.
- **RLS مفتوح على نطاق ضيق** لكل جدول جديد (`allow_all_anon` لـ anon/authenticated) بدل سياسة موحّدة على كل الجداول — قرار متعمّد لتفادي فتح الصلاحيات عشوائياً.
- **لا بيئة تجريبية (staging)** — كل تعديل يُختبر مباشرة عبر preview محلي ثم يُرفع لقاعدة الإنتاج الحية بعد تطبيق أي migration يدوياً من طرف المستخدم عبر Supabase SQL Editor (الوكيل لا يُطبّق migrations تلقائياً أبداً).

## دورة حياة الطلبية (الجزء الأهم لفهم النظام)

```
استلام الطلبية (تيليجرام/ويب)
        │
        ▼
صندوق الوارد (معالجة + شاشة مقسومة + طابور متسلسل)
        │
        ▼
طلب مبيعات (erp_sales_orders + erp_sales_order_lines)
        │  حالة: "تسجيل الطلب" → ... → "معتمد"
        ▼
عند الاعتماد: دالة توجيه واحدة approve_sales_order() توزّع كل سطر:
        │
        ├─ سطر منتج جاهز ──→ يخصم من المخزون، أو يُنشئ طلب شراء تلقائي
        ├─ سطر باب ────────→ erp_door_orders + erp_door_order_items
        │                     (قد يكون "مجرى فقط" فيبقى "قيد الاستكمال")
        ├─ ملحقات الباب ───→ ريش/ماتور: توفر أو طلب شراء تلقائي
        └─ سطر صيانة ──────→ erp_maintenance_requests
        │
        ▼ (لأسطر الباب فقط)
الإنتاج: استكمال المواصفات → احتساب هندسي → حاسبة BOM → صرف من المخزون
        │
        ▼ (عند وصول الطلبية لحالة "تم التوريد")
التركيب: سند إخراج → تقرير ميداني (صور + أوقات + مستلم) → تأكيد العميل عبر تيليجرام
        │
        ▼
"مكتملة" + تحديث حالة طلبية الباب إلى "جاهزة"
```

## الوحدات (Modules)

### 1. صندوق وارد الطلبيات (`/dashboard/sales/submissions`)
- استقبال عبر **بوت تيليجرام** (محادثة ذكية: هل العميل موجود أم جديد؟) أو **نموذج ويب**.
- كل طلبية واردة (`erp_order_submissions`) نص/صورة/تسجيل صوتي + بيانات عميل.
- **شاشة معالجة مقسومة** (`/dashboard/sales/submissions/[id]`): المحتوى الأصلي في عمود، نموذج الإدخال الفعلي في عمود آخر.
- بعد التأكيد: **طابور متسلسل** ينتقل تلقائياً لأقدم طلبية تالية بانتظار المعالجة.

### 2. المبيعات (`/dashboard/sales`)
- نموذج طلبية موحّد (`SalesOrderForm`) يدعم 4 أنواع أسطر: منتج جاهز، باب رول (+ملحقات ريش/ماتور)، صيانة، ريش/جبهة بدون باب.
- خيار **"مجرى فقط الآن"** على سطر الباب: يكفي إدخال الارتفاع، تُستكمل بقية المواصفات لاحقاً على **نفس السطر** (لا سطر جديد).
- **دالة التوجيه `approve_sales_order`**: نقطة التحول الوحيدة — عند الاعتماد تُوزَّع كل الأسطر تلقائياً للأقسام المعنية.
- جدول واحد بدل Kanban (بحث + فلتر حالة + تعديل حالة مباشر).

### 3. العملاء (`/dashboard/customers`)
- جدول عملاء بحث ذكي + اسم الشركة/المؤسسة (إن وُجد) + رابط لسجل كل عميل وطلبياته.

### 4. طلبيات الأبواب (`/dashboard/production/door-orders`)
- تُنشأ تلقائياً عند اعتماد سطر باب من المبيعات (المسار الرئيسي)، أو يدوياً من نموذج مستقل (لحالات استثنائية بدون طلب مبيعات رسمي).
- **استكمال المواصفات**: لأصناف "قيد الاستكمال" (مجرى فقط) — نموذج يُحدّث نفس السطر، مع الحفاظ على ارتفاع المجرى الأصلي كمرجع منفصل (`guides_height_mm`) عن الارتفاع الفعلي النهائي (`height_mm`)، لأنهما قد يختلفان.
- **الاحتساب الهندسي** (`door-engineering.ts`): الوزن، نوع الطاسة، نوع الخد، عدد/نوع الزنبرك — بناءً على الأبعاد ووزن الصنف بالمتر المربع.
- **حاسبة BOM** (`door-bom-calculator.ts`): تحسب كميات المواد (ماسورة، براغي، طاسة، خد، كرسي، مجرى، علاقة، ستوبات، يد باب)؛ البنود "المؤكدة" فقط (وليست الغامضة كالوجه/الأكلون/التباشيم/الجبهة) قابلة للربط بصنف حقيقي من الإعدادات وصرفها فعلياً من المخزون (`erp_bom_issues` كسجل حركة، يمنع الصرف المزدوج).

### 5. التركيب (`/dashboard/installation`) — **من أين تأتي طلبيات التركيب**
- الطلبية تظهر في طابور التركيب تلقائياً بمجرد أن يضبط أحد يدوياً حالة طلبية الباب (`erp_door_orders.status`) إلى **"تم التوريد"** من صفحة تفاصيل طلبية الباب (قائمة منسدلة الحالة) — لا يوجد إجراء تلقائي آخر يرفعها لهذه الحالة، القرار بشري بحت (بعد أن يرى الفريق أن الباب جاهز فعلياً للخروج من المصنع).
- بمجرد الوصول لحالة "تم التوريد" مع `installation_status` فارغة، تظهر في قسم "بانتظار الإخراج".
- **سند إخراج**: اسم الفريق + من قام بالإخراج → `installation_status = 'قيد التركيب'`.
- **تقرير ميداني**: صور قبل/بعد، وقت وصول/مغادرة، اسم ورقم المستلم → `installation_status = 'بانتظار تأكيد العميل'` + رسالة تيليجرام لمن قام بالإخراج بزر "✅ تأكيد استلام العميل".
- تأكيد الزر (عبر webhook البوت) → `installation_status = 'مكتملة'` + تحديث `status` الأصلي إلى "جاهزة".
- **تصميم متعمّد**: `installation_status` عمود مستقل تماماً عن `status` الأصلي — طبقة إضافية لا تُغيّر أو تُصادر معنى الحقل القديم، لتفادي كسر أي فلترة/بيانات تاريخية.

### 6. المشتريات (`/dashboard/purchasing/requests`)
- تُنشأ تلقائياً من `approve_sales_order` عند نقص مخزون (منتج أو ماتور ملحق بباب).

### 7. الصيانة (`/dashboard/maintenance/requests`)
- تُنشأ تلقائياً من سطر صيانة عند اعتماد طلب مبيعات.

### 8. الإعدادات (`/dashboard/settings`)
- هامش ربح افتراضي، قاعدة تقريب، عملة.
- **عتبات SLA** قابلة للتعديل: استكمال باب، شراء متأخر، صيانة متأخرة، تركيب متأخر، طلبية واردة متأخرة.
- **ربط مواد BOM بالمخزون**: كل بند BOM مؤكد ↔ صنف حقيقي من الكتالوج.

### 9. اللوحة التنفيذية (`/dashboard`)
- KPIs تشغيلية: استكمال أبواب، شراء معلّق، صيانة معلّقة، صندوق وارد، تركيب (بانتظار إخراج / جاري).
- **لا تحتوي بيانات مالية** (نُقلت لصفحة التقارير) — مخصصة فقط لعنق الزجاجة التشغيلي عبر الأقسام.
- تنبيهات SLA موحّدة لكل الفئات مع رابط مباشر لمكان المعالجة.

### 10. التقارير والإحصائيات (`/dashboard/reports`)
- الملخص المالي (إيرادات، مصروفات، صافي دخل)، مبيعات الشهر مقابل الشهر الماضي، الأصناف الأكثر مبيعاً، أداء الإنتاج، إجمالي المخزون.

### 11. الأجندة اليومية (`/dashboard/agenda`)
- تختار اسمك → يُطابَق دورك (`erp_staff.role`) → قائمة مصنّفة بكل ما يحتاج متابعتك، كل بند برابط مباشر لمكان التنفيذ.
- بنود المشتريات/الصيانة (بلا صفحة تفاصيل مستقلة) تفتح صفحة القائمة مع تمرير تلقائي وإبراز مؤقت للسطر المحدد (`?highlight=<id>` + مكوّن `HighlightScroll`).
- **قيد**: التصنيف حسب الدور المُختار من قائمة، وليس هوية موثّقة (بلا تسجيل دخول حقيقي).

## بوت تيليجرام (`src/app/api/telegram-webhook`)

- **استقبال طلبيات**: محادثة بحالات متسلسلة (`erp_telegram_pending_submissions`) — اختيار عميل موجود/جديد ثم إرسال المحتوى.
- **اعتماد الطلبية عن بُعد**: زر "✅ اعتماد الطلبية" بعد إدخال معالج الطلبيات لها.
- **تأكيد استلام التركيب**: زر "✅ تأكيد استلام العميل" بعد حفظ التقرير الميداني.
- جميع الصور/التسجيلات الصوتية تُرفع لـ Supabase Storage (bucket عام `order-submissions`).

## ملاحظة حول Migrations

كل migration جديد يُكتب في `supabase/migrations/` لكنه **لا يُطبَّق تلقائياً** — يجب على المستخدم نسخه وتشغيله يدوياً عبر Supabase SQL Editor، ثم تأكيد ذلك ("جاهز") قبل اعتبار الميزة المرتبطة به جاهزة للاستخدام الفعلي.
